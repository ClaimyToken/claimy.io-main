Open source, secrets elsewhere

The Angular app and Edge Function sources are meant to be forked and studied. Anything that must stay private — Supabase service role keys, wallet encryption material, relayer private keys, RPC keys with billing — belongs in Supabase Edge Function secrets or your local .env file, which is gitignored. Never paste those into the frontend bundle.

Rule of thumb: if it can move money or read every row in your database, it is not in the repo and not in environment.prod.ts as a secret — only as server-side configuration.

What you are running

Claimy is a Solana-oriented web app (Angular) that talks to Supabase (Postgres + Edge Functions) for accounts, custodial deposit addresses, playable credits, and withdrawal orchestration. Optional pieces include an Anchor on-chain vault program under programs/claimy-vault/ for PDA-based withdrawals, depending on how you configure the stack.

Layer Role
Angular SPA Wallet UX, registration/login flows, deposit modal, credits display, calls to Edge Functions over HTTPS.
Supabase Postgres Users, nonces, referral data, playable credit ledger — applied via SQL migrations in docs/migrations/.
Edge Functions Register/login, credits sync, game sessions (Flowerpoker + Blackjack + Dice), Playhouse feed/ranking stats, withdrawals, referrals.
Solana RPC Read balances, send SPL transfers from your vault / relayer — configured in Edge secrets + local .env for the app.

Prerequisites

  • Node.js compatible with Angular 16 (LTS 18.x or 20.x is a safe choice).
  • npm at the repo root for npm install.
  • A Supabase project where you can run SQL and deploy Edge Functions.
  • An SPL token mint on the same cluster as your RPC (mainnet vs devnet — stay consistent).
  • Phantom (or compatible) for the same cluster when testing wallet flows in the browser.

For building the on-chain program, see Anchor + Rust; on Windows, many teams use WSL2 for Solana tooling. The web app itself runs fine on Windows with Node alone.

Clone and install

git clone <your-fork-or-upstream-url>
cd claimy.io-main   # or your repo folder name
npm install

Local environment (.env)

Copy env.template to .env in the project root (next to package.json). Fill in at least:

  • CLAIMY_SUPABASE_URL — your project URL from Supabase (no trailing slash).
  • CLAIMY_SOLANA_RPC_URL — HTTPS JSON-RPC for the same cluster as your mint.
  • CLAIMY_SPL_MINT — base58 mint address of your SPL token (replace Claimy’s mint when experimenting).

Run npm run env:sync (or let npm start run it first). That regenerates src/environments/env.overrides.ts so Angular reads your values during ng serve. The start script in package.json is: sync env, then serve. 

npm run env:sync
npx ng serve
# open http://localhost:4200/
Production builds use environment.prod.ts — not .env. Set public URLs and mint there in CI/CD or your host’s config. See docs/FRONTEND_AND_ENV.md.

Supabase: database then functions

Schema and migrations

In Supabase SQL Editor, run the schema and migrations from the repo in a sensible order. The handoff checklist starts with docs/supabase-schema.sql, then add tables from docs/migrations/ (for example playable credits, referrals, deposit snapshots) as your feature set requires. Always review DROP / destructive blocks before running against anything important.

Edge Function secrets

In Project Settings → Edge Functions → Secrets, configure everything your functions need: Supabase URL, service role, Solana RPC, SPL mint, encryption key for deposit wallets, relayer keys for withdrawals, etc. The full list and semantics are documented in docs/EDGE_FUNCTIONS.md and docs/SUPABASE_SETUP.md.

Deploy functions

Deploy each function (Dashboard paste or supabase functions deploy) so the slug matches what the Angular app calls. URLs are always: 

{{ '{' }}supabaseUrl{{ '}' }}/functions/v1/{{ '{' }}slug{{ '}' }}

Slugs include check-username, register-phantom, wallet-login, account-linked-wallet, claimy-credits, claimy-profile, withdraw-spl, claimy-referrals, flowerpoker-game, blackjack-game, dice-game, playhouse-feed, admin-sweep-wallets, and others listed in supabase/functions/README.md and docs/README.md.

Playhouse feed

Public feed and “My bets” both POST to Edge playhouse-feed with {{ '{' }} "action": "list_bets", "page", "pageSize" {{ '}' }} and optional walletAddress for the filtered view. The function calls RPC playhouse_list_settled_bets with the service role — run docs/migrations/claimy_playhouse_feed.sql and claimy_playhouse_player_ranking_stats.sql + claimy_playhouse_include_blackjack.sql (RPC playhouse_player_ranking_stats for action player_ranking_stats) before deploying. In supabase/config.toml, set verify_jwt = false under [functions.playhouse-feed] so the browser can call it with anon headers only (same pattern as other public Edge endpoints).

npx supabase@latest functions deploy playhouse-feed --project-ref <ref>

If Playhouse feed fails with browser Failed to fetch and Network shows OPTIONS /functions/v1/playhouse-feed → 404, the function slug is not deployed on that project yet. Deploy exactly playhouse-feed (CI job names can differ; the URL path must not).

How the main flows work

Registration

The register flow checks username availability, then calls register-phantom, which creates a custodial deposit keypair for the user, encrypts the secret with a server-only key, and stores the public address and ciphertext in Postgres. Users send SPL to that deposit address; the app never holds the private key in the browser.

Login

wallet-login returns session data for the connected wallet (username, deposit address, referral fields, optional provably-fair gamesClientSeed, etc.). The UI uses this to show “Your wallet” and credits. Linked-wallet verification uses account-linked-wallet as documented in the repo.

Credits and deposits

Playable balance lives in the database and is reconciled with on-chain SPL on the custodial deposit ATA via claimy-credits actions such as sync_from_chain. The deposit modal in the app shows the deposit address; after you send tokens on-chain, refresh/sync updates the displayed balance when your backend is configured correctly.

Withdrawals

Withdrawals are initiated from the app but executed by withdraw-spl using server-side signing keys. That can be a hot wallet (“simple vault”) or a PDA program path, depending on your deployment and docs/CLAIMY_VAULT.md. Fund the vault / relayer with SOL for fees and SPL tokens as needed.

Admin sweep operations

Account Settings now includes a whitelist-gated admin panel powered by admin-sweep-wallets for custodial deposit-wallet consolidation. It supports summary_only (scan only), dry_run (selection + logs, no transfers), and execute (transfers + logs), plus top-holder limiting and debug trace output.

Required secrets: CLAIMY_ADMIN_WALLETS, CLAIMY_SWEEP_FEE_PAYER_PRIVATE_KEY, DEPOSIT_WALLET_ENCRYPTION_KEY, and optional default destination CLAIMY_SWEEP_DESTINATION_WALLET. Destination precedence is request override -> default destination secret -> caller wallet.

The fee payer wallet must hold SOL for transaction costs. Sweep execution also updates deposit_chain_balance_snapshot so credits reconciliation remains correct after funds are moved out.

Games and provably fair checks

Claimy game pages use Edge game handlers that lock stake, persist session metadata in claimy_game_sessions, and settle through claimy_apply_credit_delta. Flowerpoker, Blackjack, and Dice follow the same pattern (Dice uses one-shot roll with HMAC label claimy-dice|v2|… and rollSpace 1000 in the fairness snapshot). The UI includes browser-side verification utilities so players can compare committed seed/hash values against revealed settlement data.

Ranking now refreshes right after game settlement, and live nav toasts announce rank-ups (including multi-tier jumps) with the updated RTP/payout multiplier.

Using your own SPL token

Point CLAIMY_SPL_MINT (and Supabase secrets) at your mint. Create a mint on the cluster you use, note decimals, fund user ATAs and vault ATAs as required, and ensure the same mint appears in Edge secrets and the Angular env. Mismatch between cluster, mint, or RPC is the most common source of “nothing syncs” or “withdraw fails” in local experiments.

Security checklist

  • Do not commit .env or service role keys.
  • Do not put relayer keys or encryption secrets in Angular environments.
  • Rotate RPC keys if they leak; treat browser-exposed RPC URLs as public.
  • Review Supabase RLS and function policies for your production project.
